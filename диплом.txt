ВВЕДЕНИЕ
     В рамках данной ВКР рассматривается разработка веб-приложения Vulnerability Checker, предназначенного для автоматизированного анализа зависимостей проектов на наличие известных уязвимостей с последующим формированием отчётов в формате PDF.
     Актуальность проекта обусловлена необходимостью обеспечения безопасности программного обеспечения при использовании сторонних библиотек и фреймворков. Современные тенденции разработки требуют от специалистов инструментов для контроля качества и безопасности используемых компонентов.
     Целью проекта является создание веб-приложения, позволяющего автоматизировать проверку зависимостей проекта на наличие уязвимостей и генерацию структурированных отчётов о результатах анализа.
     Для достижения этой цели необходимо решить следующие задачи: – анализ предметной области существующих решений в области аудита безопасности зависимостей; – формулировка функциональных, нефункциональных и пользовательских требований к системе; – моделирование бизнес-процессов и данных системы; – выбор архитектурных решений и технологического стека с обоснованием применимости каждого инструмента; – проектирование архитектуры приложения; – реализация программных компонентов в соответствии с требованиями безопасности; – тестирование и верификация работоспособности всех функций системы; – разработка документации для пользователей и разработчиков.
     Особенность проекта заключается в том, что система предоставляет локальное решение с веб-интерфейсом, не требующее подписки на облачные платформы и позволяющее экспортировать результаты анализа в формате PDF.
     Практическая значимость работы состоит в создании инструмента, который делает безопасность естественной и быстрой частью процесса разработки, а не его тормозом. Работа состоит из введения, трёх разделов, заключения, списка использованных источников и приложений.
     В аналитическом разделе проведён анализ предметной области и существующих продуктов, сформулированы требования к системе, спроектирована архитектура приложения и составлены пользовательские варианты использования.
     
     В проектно-конструкторском разделе приведён выбор средств разработки, проектирование архитектуры клиента и сервера, описан процесс реализации системы.
     В экспериментальном разделе приведены результаты тестирования основных функций системы.
     
     
     
     
     
     
     
     
     
     
     
     
     
     
     
     
     
     
     
     
     
     
     
     1 Аналитический раздел

     1.1.1  Анализ предметной области

     Использование сторонних библиотек и фреймворков стало стандартом де-факто для ускорения процессов создания приложений. Разработчики активно применяют open-source пакеты для реализации уникальной бизнес-логики вместо повторного решения уже существующих задач. Однако такая зависимость от внешних компонентов порождает серьёзные проблемы безопасности, требующие профессионального подхода к решению.
     Только за 2024 год было зафиксировано 217 успешных кибератак, при этом каждая пятая обнаруженная уязвимость носит критический характер. Регуляторы последовательно ужесточают требования к безопасной разработке программного обеспечения, что делает автоматизацию аудита зависимостей не просто удобным инструментом, а необходимой частью процесса.
     Веб-приложение предназначено для следующих категорий пользователей:
     * DevSecOps-инженеры - специалисты по безопасности, отвечающие за контроль и мониторинг уязвимостей в проектах;
     * Разработчики - разработчики серверной логики, использующие Python-пакеты для реализации бизнес-функций;
Предметной областью проекта является программное обеспечение, которое:
     1 Обеспечивает клиентский интерфейс для:
     * загрузки файлов зависимостей проектов
     * просмотра результатов анализа уязвимостей
     * управления настройками и API ключами пользователя.
     2 Реализует серверную часть для: 
     * хранения данных пользователей и результатов сканирования; 
     * взаимодействия с внешними API сервисов проверки CVE; 
     * преобразования и передачи результатов анализа.
Разработанный веб-приложение оптимизирует процесс контроля безопасности зависимостей проекта и упрощает документирование результатов проверок через экспорт отчётов в PDF.
     1.2 Обзор существующих решений
     1.2.1 Классификация инструментов аудита уязвимостей

     На современном рынке представлено множество инструментов для сканирования уязвимостей различных типов, которые можно разделить на следующие категории:
     * SaaS-платформы (Snyk, Meterian), облачные решения с расширенным функционалом
     * Локальные CLI-инструменты (pip-audit, npm audit), утилиты командной строки для локального использования
     * Интегрированные решения (Dependabot, Renovate), инструменты, работающие в связке с системами управления версиями
     1.2.2 Детальный анализ существующих решений

     Snyk является одной из наиболее известных коммерческих платформ для анализа безопасности кода и зависимостей. Решение предлагает комплексный подход к сканированию уязвимостей как в исходном коде, так и в сторонних библиотеках. Платформа поддерживает множество языков программирования включая JavaScript, Java, Python, Ruby и другие. Основное преимущество Snyk заключается в обширной базе данных уязвимостей и возможности автоматического исправления проблем через интеграцию с системами CI/CD. Однако существенным недостатком является зависимость от облачного API, что делает невозможным использование решения для проектов с особыми требованиями к конфиденциальности. Кроме того, коммерческая модель лицензирования может быть препятствием для небольших организаций и стартапов.
     Meterian представляет собой ещё одну SaaS-платформу для мониторинга безопасности зависимостей. Решение отличается интуитивно понятным интерфейсом и возможностью отслеживания обновлений пакетов в режиме реального времени. Платформа предоставляет уведомления о новых уязвимостях в используемых библиотеках, а также предлагает рекомендации по обновлению до безопасных версий. Meterian поддерживает интеграцию с GitLab, GitHub и другими системами контроля версий, что обеспечивает автоматизацию процессов аудита. Основным ограничением является необходимость постоянного подключения к интернету для получения актуальных данных о уязвимостях, что может быть критично в условиях ограниченного доступа к внешним ресурсам или при работе с закрытыми контурами информационных систем.
     Safety - это открытый инструмент командной строки, разработанный специально для анализа Python-проектов на наличие известных уязвимостей в зависимостях. Инструмент работает локально и не требует подключения к облачным сервисам, что делает его удобным для работы с конфиденциальными проектами. Safety поддерживает чтение файлов requirements.txt и Poetry lock-файлов, извлекая список пакетов и их версий для последующей проверки. Основным преимуществом является открытая лицензия и возможность запуска в любой среде включая офлайн. Недостатком является текстовый формат вывода без визуализации результатов, а также отсутствие встроенного механизма формирования отчётов в формате PDF для документирования.
     Dependabot - это инструмент автоматического обновления зависимостей от GitHub, который мониторит проекты на предмет появления новых версий библиотек и создаёт pull requests с обновлениями. Решение глубоко интегрировано в экосистему GitHub и поддерживает множество языков программирования. Основным преимуществом является автоматизация процесса обновления и минимизация ручного труда разработчиков. Однако Dependabot ограничен платформой GitHub, что делает его неприменимым для проектов, размещённых на других системах контроля версий или хранилищах кода.
     1.2.3 Выявленные недостатки существующих решений

     Snyk требует коммерческой подписки для расширенных функций, работает через облачный API, что не всегда допустимо при работе с конфиденциальными проектами внутри организаций.
     Meterian аналогично Snyk зависит от облачных сервисов и требует постоянного подключения к интернету, что может быть критичным в условиях ограниченного доступа или работы с закрытыми контурами.
     Dependabot интегрирован исключительно в экосистему GitHub, ограничивает выбор платформы разработки.
     Safety предоставляет только текстовый вывод без структурированных отчётов и визуального представления результатов.
     1.2.4 Обоснование разработки нового продукта

     Разработка Vulnerability Checker позволяет решить следующие проблемы:
     
     * предоставление локального решения с возможностью работы без внешних API;
     * создание удобного веб-интерфейса для визуализации результатов;
     * поддержка экспорта отчётов в формате PDF для документирования;
     * возможность расширения функционала под конкретные потребности пользователей.

     2 Проектно-конструкторский раздел 

     2.1 Анализ технического задания на проектирование и выбор средств и методов решения задачи 

     2.1.1 Выбор языка программирования для разработки клиента и сервера 

     Для реализации клиентской части веб-приложения в качестве основного языка программирования выбран TypeScript. Данный выбор обусловлен несколькими ключевыми факторами: TypeScript является строго типизированным надмножеством JavaScript, что позволяет выявлять ошибки на этапе компиляции и повышает надёжность кода, широкая экосистема инструментов разработки для TypeScript обеспечивает эффективный процесс создания крупных приложений, поддержка JSX-синтаксиса позволяет комфортно работать с библиотекой React. 
     В качестве основной библиотеки для построения пользовательского интерфейса выбран React. Данный фреймворк обеспечивает декларативный подход к разработке, компонентную архитектуру и эффективный механизм обновления DOM через виртуальное представление. Использование TypeScript в сочетании с React позволяет создавать типобезопасные компоненты с поддержкой интеллектуального автозаполнения в средах разработки. 
     Для сборки проекта применяется инструмент Vite, который обеспечивает быстрый Hot Module Replacement (HMR) и оптимизированный билд для производственной среды.
     
     2.1.2 Выбор библиотек и фреймворков

     При создании клиентской части приложения были проанализированы доступные инструменты для маршрутизации навигации, работы с HTTP-запросами и визуального оформления интерфейса. Для организации клиентской маршрутизации выбран react-router-dom, предоставляющий декларативный способ определения путей и управления состоянием навигации без перезагрузки страницы. Данный пакет обеспечивает историю переходов между экранами приложения и поддерживает вложенные маршруты, что необходимо для структуры веб-приложения с множеством функциональных разделов.
     Для осуществления HTTP-запросов к серверной части выбран axios. Библиотека предоставляет promise-based API для выполнения сетевых запросов, поддерживает автоматическую десериализацию JSON ответов, настройку заголовков авторизации и возможность создания интерсепторов для перехвата запросов и ответов. Эти возможности особенно важны для приложения, работающего с внешними системами поиска уязвимостей.
     Для визуального оформления интерфейса применена библиотека Tailwind CSS. Данный фреймворк предлагает utility-first подход к стилизации, позволяя определять внешний вид элементов непосредственно в HTML-разметке через классовые утилиты. Такой подход ускоряет разработку компонентов и обеспечивает консистентность дизайна во всём приложении.
     Для работы с иконками используется пакет react-feather, предоставляющий набор SVG-иконок в виде React-компонентов. Данный выбор обусловлен лёгкостью интеграции, минимальным размером пакета и единообразием стиля визуальных элементов.
     Генерация отчётов о результатах поиска уязвимостей реализована с помощью библиотеки pdfmake. Данная библиотека позволяет создавать PDF-документы программно на стороне клиента, формируя структурированные документы с таблицами результатов и текстовыми описаниями найденных уязвимостей.
     2.1.3 Выбор среды разработки

     В качестве основной среды разработки для клиентской части выбран VS Code. Данный выбор обусловлен широкими возможностями по настройке и расширению функциональности, а также поддержкой множества языков программирования и фреймворков через систему расширений. Важными расширениями в процессе разработки стали ESLint и Prettier. Первое обеспечивает статический анализ кода на соответствие стандартам качества, второе автоматизирует форматирование исходного текста согласно единому стилю оформления. Подобный подход позволяет выявить потенциальные проблемы ещё до компиляции проекта и поддерживать единообразие кода при работе с проектом.
     
     2.1.4 Выбор операционной системы

     Для развёртывания серверной части выбран дистрибутив Ubuntu Linux, предлагающий надёжную и защищённую операционную систему с длительным сроком поддержки стабильных релизов. Данный выбор обусловлен преимуществами перед Windows Server в контексте веб-разработки: открытые исходные коды большинства компонентов экосистемы, оптимизированное управление ресурсами при высоких нагрузках и богатый выбор инструментов контейнеризации.
     Для организации оркестрации сервисов применяется Docker с использованием docker-compose для управления многоконтейнерными приложениями. Такое решение позволяет изолировать зависимости бэкенда (Python 3.x, FastAPI, SQLModel), базы данных PostgreSQL от операционной системы хоста и обеспечить воспроизводимость среды развёртывания на любых машинах. Контейнеры сервера и клиента запускаются в едином docker-compose.yml файле, что упрощает процесс деплоя.
     
     2.1.5 Выбор СУБД

     В качестве системы управления базами данных выбрана PostgreSQL 17 - реляционная СУБД с открытым исходным кодом, признанная одной из наиболее надёжных решений для хранения структурированных данных. Данный выбор обусловлен поддержкой сложных SQL-запросов, транзакционной целостностью и расширенными возможностями индексирования, необходимыми для эффективного поиска уязвимостей по множеству параметров одновременно.
     Сравнение PostgreSQL с альтернативами показывает его преимущества для данного проекта. SQLite не подходит из-за отсутствия встроенной поддержки сетевых соединений и ограничений многопользовательского доступа через блокировку файла. MySQL, обладая хорошей совместимостью, уступает в возможностях индексации JSON-полей и экосистемы расширений. MongoDB обеспечивает слабую согласованность данных при связанных операциях, что критично для хранения уязвимостей с множеством внешних ключей между таблицами Package, PackageVersion и Vulnerability.
     Для работы с ORM применена библиотека SQLModel, объединяющая возможности Pydantic для валидации данных и SQLAlchemy для управления связями между таблицами. Миграции схемы базы данных осуществляются через Alembic - независимый инструмент управления изменениями структуры БД, позволяющий версионировать эволюцию схем вместе с кодовой базой приложения.

     2.7 Проектирование базы данных

     Важным этапом в создании программного обеспечения является проектирование базы данных. Оно задаёт структуру и порядок хранения данных.
     2.7.1 Концептуальная схема

     Концептуальная модель представляет собой высокоуровневое описание предметной области. Она определяет ключевые сущности системы и взаимосвязи между ними.
     Основные сущности системы:
* Пользователь (User) – содержит учетные данные, личную информацию и API ключи зарегистрированных пользователей системы;
* API ключ (Api key) – служит для аутентификации внешних вызовов серверного API;
* Пакет (Package) – хранит данные о библиотеках и фреймворках, доступных в официальном репозитории;
* Версия пакета (Package version) – представляет собой конкретную версию библиотеки с указанием номера релиза;
* Уязвимость (Vulnerability) – содержит информацию о известных CVE для сравнения версий пакетов;
* Компания (Company) – организует пользователей в рамках корпоративной структуры.
     Связь между сущностями отображена на рисунке 4.

Рисунок 4 – Схема базы данных
     Уязвимость (Vulnerability) содержит информацию о известных CVE для сравнения версий пакетов. Таблица спроектирована с использованием денормализованного подхода: вместо внешних ключей к таблицам Package и PackageVersion используются текстовые поля language, library_name и library_version. Такой выбор обусловлен тем, что связь между уязвимостью и пакетом является семантической, а не референциальной — каждая запись представляет информацию о конкретной версии библиотеки. Кроме того, механизм сопоставления версий требует строковых операций сравнения (с учётом диапазонов уязвимых версий), которые невозможно выразить через реляционные ограничения базы данных. Денормализация также обеспечивает более быстрый доступ к полям.
     Пользователь системы может имепть несколько API ключей для доступа к серверному API Vulnerability Checker, при этом каждый ключ привязан ровно к одному владельцу. Связь реализована через промежуточную таблицу UserAPIKeyLink, содержащую составной первичный ключ из user_id и api_key_id.
     Каждый пакет может иметь множество версий, при этом каждая версия принадлежит строго определённому пакету. Связь обеспечивается внешним ключом package_id в таблице PackageVersion, а также уникальным индексом (package_id, version) для предотвращения дублирования версий одного пакета.
     Система поддерживает корпоративное использование через организацию пользователей в рамках компаний. Каждая компания может содержать множество зарегистрированных пользователей, и пользователь при этом принадлежит одной компании. Связь дополнена полем level, определяющим роль пользователя внутри организации (администратор или сотрудник).
     2.8 Разработка алгоритмов работы программы

     После проектирования базы данных и выбора технологий был реализован функциональный слой приложения, включающий ключевые алгоритмы обработки пользовательских запросов. Этот раздел описывает основные алгоритмические решения, обеспечивающие работу системы: обновление JWT-токенов для поддержания сеанса аутентификации, обработку загрузки файлов зависимостей через drag-and-drop интерфейс и генерацию отчётов о результатах сканирования в формате PDF.
     2.8.1 Алгоритм обновления токена
     
     Механизм обновления токена обеспечивает непрерывную работу приложения при истечении срока действия access token. Архитектурно этот алгоритм реализован через interceptors библиотеки axios, что позволяет прозрачно обрабатывать ситуации аутентификации без изменения бизнес-логики компонентов.
     Алгоритм работает следующим образом: когда сервер возвращает HTTP-код 401 Unauthorized (токен истёк), interceptor перехватывает ошибку и пытается получить новую пару токенов через endpoint /token/refresh, передавая refresh token в теле запроса. При успешном обновлении новый access token сохраняется как в localStorage, так и устанавливается как заголовок Authorization для текущего запроса, который затем повторяется. Если обновление не удалось (например, истёк refresh token), пользователь перенаправляется на страницу логина, а данные аутентификации очищаются.
       // Функция для получения нового access-токена с использованием refresh-токена
       const refreshAccessToken = async (): Promise<string> => {
           // Получаем refresh-токен из локального хранилища
           const refreshToken = localStorage.getItem('refreshToken');
           // Если refresh-токена нет, выбрасываем ошибку
           if (!refreshToken) throw new Error('No refresh token available');
           try {
               // Отправляем запрос на сервер для обновления токенов
               const response = await axios.post(`${apiUrl}/token/refresh`, {
                   refresh_token: refreshToken,
               });
               // Извлекаем новые токены из ответа
               const { access_token, refresh_token: newRefreshToken } = response.data; 
               // Сохраняем оба токена в localStorage
               localStorage.setItem('authToken', access_token);
               localStorage.setItem('refreshToken', newRefreshToken); 
              // Обновляем заголовок Authorization во всех HTTP-запросах
               updateAxiosInstance(access_token); 
               // Возвращаем новый access-токен
               return access_token;
           } catch (error) {
               // При ошибке очищаем все токены и перенаправляем на страницу входа
               console.error('Failed to refresh token:', error);
               localStorage.removeItem('authToken');
               localStorage.removeItem('refreshToken');
               window.location.href = '/login';
               throw error;
           }
       };
     Флаг _retry в объекте конфигурации запроса предотвращает бесконечную рекурсию при повторных попытках обновления. При первой ошибке 401 устанавливается флаг _retry = true, после чего выполняется обновление токена и повторный запрос. Если обновление также завершается неудачей, например, истёк refresh token, пользователь перенаправляется на страницу логина.
       // Настройка interceptor для автоматической обработки ошибок 401
       const setupInterceptors = (instance: typeof api) => {
           instance.interceptors.response.use(
               // Если запрос успешен, возвращаем ответ как есть
               (response) => response, 
               // При ошибке проверяем, связана ли она с истекшим токеном
               async (error) => {
                   const originalRequest = error.config as AxiosRequestConfig & {_retry?: boolean}; 
                   // Если ошибка 401 и запрос еще не был повторен
                   if (error.response?.status === 401 && !originalRequest._retry) {
                // Помечаем запрос как повторяемый, чтобы избежать рекурсии
                       originalRequest._retry = true; 
                       try {
                           // Получаем новый токен
                           const newToken = await refreshAccessToken(); 
                    // Обновляем заголовок Authorization в оригинальном запросе
                           originalRequest.headers!.Authorization = `Bearer ${newToken}`; 
                           // Повторяем оригинальный запрос с новым токеном
                           return instance(originalRequest);
                       } catch (refreshError) {
                           // Если обновление не удалось, возвращаем ошибку
                           return Promise.reject(refreshError);
                       }
                   } 
                   // Для других ошибок просто возвращаем ошибку
                   return Promise.reject(error);
               },
           );
       };
       // Применяем interceptor к экземпляру axios
       setupInterceptors(api);
     2.8.2 Алгоритм обработки загрузки файла
     
     Обработка файлов зависимостей включает несколько этапов: визуальную обработку drag-and-drop событий, валидацию загруженного файла и парсинг содержимого для извлечения списка пакетов и версий. Фронтенд-компонент FileUpload реализует интерфейс взаимодействия с пользователем, а хук useFileProcessor - логику обработки данных.
     Алгоритм drag-and-drop начинается с событий onDragOver и onDragLeave, которые управляют визуальным состоянием зоны сброса. При событии onDrop извлекается первый файл из event.dataTransfer.files и передаётся в функцию processFile для дальнейшей обработки.
       // Обработчик события наведения файла над зоной сброса
       const handleDragOver = (event: DragEvent<HTMLDivElement>): void => {
           event.preventDefault(); // Предотвращаем стандартное поведение браузера
           if (!disabled) setIsDragging(true); // Подсвечиваем зону при перетаскивании
       };
       // Обработчик события ухода файла из зоны сброса
       const handleDragLeave = (event: DragEvent<HTMLDivElement>): void => {
           event.preventDefault();
           setIsDragging(false); // Убираем подсветку при уходе курсора
       };
       // Обработчик события сброса файла в зону
       const handleDrop = (event: DragEvent<HTMLDivElement>): void => {
           event.preventDefault();
           setIsDragging(false);
           if (disabled) return; 
           const files = event.dataTransfer.files; // Получаем список файлов из drag-события
           if (files.length > 0) processFile(files[0]); // Обрабатываем первый файл
       };
     
     2.8.3 Алгоритм генерации PDF-отчёта
     
     Генерация отчётов о результатах сканирования реализуется через библиотеку pdfmake, которая позволяет создавать PDF-документы на стороне клиента с использованием declarative JSON-формата описания структуры документа. Алгоритм включает три основных этапа: формирование статистической сводки, построение таблицы обнаруженных уязвимостей и добавление полных списков проверенных пакетов.
     Первый этап алгоритма вычисляет метрики сканирования: общее количество проверенных пакетов, количество уязвимых пакетов и суммарное число обнаруженных CVE-уязвимостей. Эти данные формируют заголовок отчёта.
       // Функция для вычисления общей статистики сканирования
       const calculateSummary = (packages: ReceivedPackage[]) => {
           // Отделяем уязвимые пакеты от безопасных
           const vulnerablePackages = packages.filter(
               (pkg) => pkg.cve_list && pkg.cve_list.length > 0,
           );
           
           return {
               totalPackages: packages.length, // Общее количество проверенных пакетов
               vulnerablePackages: vulnerablePackages.length, // Количество уязвимых пакетов
               totalVulnerabilities: packages.reduce(
                   (acc, pkg) => acc + (pkg.cve_list ? pkg.cve_list.length : 0),
                   0,
               ), // Суммарное число всех обнаруженных CVE
               scanDate: new Date().toLocaleDateString('ru-RU'), // Дата сканирования
           };
       };
       
     2.9 Безопасность веб-приложения

     Безопасность клиентского приложения является важным аспектом разработки, особенно при работе с аутентификационными данными и конфиденциальной информацией пользователей. Веб-приложение реализует многоуровневую стратегию защиты, охватывающую хранение токенов, передачу данных, защиту от межсайтового скриптинга (XSS) и валидацию входных данных.
     2.9.1 Хранение токенов аутентификации

     Access token и refresh token хранятся в локальном хранилище браузера. При получении ответа сервера со статусом 401 Unauthorized интерсептор axios не выполняет немедленный выход пользователя, а сначала пытается обновить access token через вызов эндпоинта /token/refresh с использованием refresh токена. Если обновление токена успешно, исходный запрос выполняется повторно с новым токеном без прерывания пользовательской сессии. При неудачном обновлении, когда refresh токен также истёк или невалиден, происходит принудительное завершение сессии - очистка обоих токенов из локального хранилища и перенаправление пользователя на страницу входа. Дополнительно logout пользователь может инициировать вручную через кнопку выхода в интерфейсе профиля.
     2.9.2 Защита от XSS-атак

     React по умолчанию экранирует все вставляемые значения через JSX, предотвращая выполнение произвольного HTML или JavaScript кода, содержащегося в данных сервера. Например, описание уязвимости, полученное с сервера и выведенное через {description}, автоматически экранируется и не может быть исполнено как код. Дополнительно при генерации PDF-отчётов с помощью библиотеки pdfmake применяется механизм рендеринга текста без интерпретации содержимого как HTML.
     2.9.3 Валидация данных на клиенте

     Валидация входных данных на стороне клиента представляет собой вспомогательный механизм предварительной проверки форм перед отправкой запроса на сервер. Так, поля логина и пароля страницы входа имеют соответствующие атрибуты для базовой проверки формата вводимых значений, то же относится и к полям формы регистрации. 
     Валидация содержимого загружаемых файлов представляет собой отдельный этап обработки перед отправкой данных на сервер. При выборе файла хук useFileProcessor считывает его содержимое в виде текстовой строки и выполняет парсинг для извлечения информации о пакетах и их версиях. Парсинг выполняется регулярным выражением /^([a-zA-Z0-9_-]+)==([a-zA-Z0-9._-]+)$/, которое распознаёт формат имя_пакета==версия (используемый в файлах requirements.txt для Python). Извлечённые данные преобразуются в структуру JSON - объект с полем packages, содержащим словарь пар «название пакета ? версия». Эта структура соответствует формату, ожидаемому серверным API. Обработка ошибок чтения файла включает очистку состояния и сброс данных при некорректном содержимом или невозможности чтения файла.
     2.10 Производительность веб-приложения

     Производительность веб-приложения влияет как на пользовательский опыт, так и на эффективность разработки. Используется несколько стратегий оптимизации на различных этапах жизненного цикла.
     Виртуальный DOM минимизирует манипуляции с реальным DOM браузера. При изменении состояния компонента React сначала вычисляет разницу между текущим и новым виртуальным представлением, а затем выполняет минимально необходимый набор операций обновления только тех элементов, которые фактически изменились. Это существенно снижает нагрузку на браузер при динамическом обновлении списков результатов сканирования уязвимостей.

     3.3.1 Описание основных характеристик и особенностей программы

     Программа работает по архитектуре Single Page Application (SPA) где веб-клиент взаимодействует с серверным API посредством RESTful HTTP запросов. Для внешних вызовов сканирования пакетов предусмотрен механизм X-API-Key авторизации который автоматически генерируется при создании аккаунта пользователя. Серверная часть использует асинхронный фреймворк FastAPI что обеспечивает высокую производительность обработки запросов благодаря поддержке синтаксиса async/await встроенной валидации данных через библиотеку Pydantic и автоматической генерации документации OpenAPI доступной через Swagger UI интерфейс.
     3.3.2 Структура приложения

     Структура программного продукта включает несколько основных компонентов, расположенных в соответствующих директориях проекта. Клиентская часть расположена в директории frontend где находится исходный код на TypeScript собранный с помощью сборщика Vite. Директория src содержит основной код приложения включая файлы App.tsx и main.tsx являющиеся точками входа. Поддиректория components содержит компоненты пользовательского интерфейса, разделённые на страницы (Landing, Login, Register, Home, Profile) и UI элементы (NavBar, CveCard, FileUpload). Директория hooks содержит кастомные хуки для управления состоянием приложения включая useAuth для аутентификации useFileProcessor для обработки загружаемых файлов и usePackageCheck для выполнения запросов сканирования. Поддиректория interface содержит TypeScript типы и интерфейсы, определяющие структуру данных, передаваемых между компонентами и сервером.
     Серверная часть расположена в директории backend где находится основной модуль main.py запускающий FastAPI приложение. Директория api содержит API роутеры для аутентификации пользователей (auth.py) управления учётными записями (users.py) и обработки запросов сканирования уязвимостей (vulnerabilities.py). Модуль models.py определяет SQL модели таблиц базы данных используя ORM SQLModel включающий таблицы User, ApiKey, Package, PackageVersion, Vulnerability и Company. Директория alembic содержит миграционные скрипты для управления схемой базы данных позволяя обновлять структуру таблиц без потери существующих данных.
     Служба парсинга уязвимостей расположена в директории package_scraper где находится основной модуль main.py отвечающий за извлечение информации о CVE из внешних источников и сохранение результатов в базу данных PostgreSQL. Все три компонента приложения (frontend, backend, package_scraper) контейнеризируются с помощью Docker для обеспечения идентичности среды выполнения на всех этапах разработки и тестирования.
     3.3.3 Обращение к приложению и описание процедур вызова

     Обращение к программе осуществляется через HTTP API эндпоинты серверной части FastAPI. Для регистрации нового пользователя необходимо выполнить POST запрос к эндпоинту /api/register передавая объект содержащий email пароль имя фамилию и опционально отчество. При успешном создании учётной записи генерируется API ключ автоматически сохраняемый в базе данных.
     Для аутентификации пользователя требуется отправить POST запрос на эндпоинт /token с данными в формате application/x-www-form-urlencoded содержащими поле username (email) и password. В случае успешной аутентификации сервер возвращает JSON объект с access_token, refresh_token и token_type. Клиентское приложение сохраняет оба токена в localStorage браузера после чего добавляет заголовок Authorization Bearer к последующим запросам защищённых ресурсов.
     Для проверки пакетов на уязвимости необходимо выполнить POST запрос к эндпоинту /api/check-packages передавая JSON объект с полем packages содержащим словарь пар название_пакета ? версия. Запрос должен включать заголовок X-API-Key со значением API ключа пользователя. Сервер сравнивает указанные версии пакетов с базой данных уязвимостей и возвращает список обнаруженных CVE идентификаторов вместе с описанием опасности каждой уязвимости.
     Для получения информации о профиле пользователя следует отправить GET запрос к эндпоинту /api/profile с заголовком Authorization Bearer содержащим access token. Сервер возвращает объект с данными пользователя включая email имя фамилию отчество номер телефона и список связанных API ключей.
     3.3.4 Входные и выходные данные

     Входными данными для приложения являются пакеты Python в формате словарь где ключ представляет название пакета а значение - его версию. При загрузке файла клиентское приложение парсит содержимое текстового файла используя регулярное выражение /^([a-zA-Z0-9_-]+)==([a-zA-Z0-9._-]+)$/ распознающее формат имя_пакета==версия применяемый в файлах requirements.txt для Python. Извлечённые данные преобразуются в структуру JSON с полем packages содержащим словарь пар название пакета и его версия после чего отправляется на сервер для проверки.
     Выходными данными приложения является список объектов содержащих информацию об обнаруженных уязвимостях каждый из которых включает идентификатор CVE название библиотеки версию библиотеки описание уязвимости уровень опасности в текстовом формате и числовую оценку по шкале CVSS. На стороне клиента эти данные отображаются в виде карточек с цветовой индикацией: красным цветом для критических уязвимостей оранжевым для высоких жёлтым для средних и зелёным для низких уровней опасности. При необходимости пользователь может экспортировать полученные результаты в формат PDF используя библиотеку pdfmake которая формирует документ содержащий заголовок отчёт даты информацию о пользователе таблицу уязвимостей с полным описанием каждой обнаруженной проблемы.
