===============================================================================
           МНОГОПРОЦЕССНАЯ СИСТЕМА ВАННЫХ КОМНАТ — ТЕОРИЯ И РАЗБОР
===============================================================================

Цель: объяснить принципы работы многопроцессной программы управления доступом 
к разделяемому ресурсу (ванная комната) с использованием:
- разделяемой памяти,
- межпроцессной синхронизации,
- механизма приоритетов ("стрик").

-------------------------------------------------------------------------------
1. ОБЩАЯ АРХИТЕКТУРА ПРОГРАММЫ
-------------------------------------------------------------------------------

Программа моделирует доступ студентов (мужчин и женщин) к общей ванной комнате 
в общежитии университета. Основные правила:

• В ванной одновременно может находиться до CAPACITY человек.
• Внутри могут быть только люди одного пола (никаких смешанных групп).
• При входе определяется приоритет через "стрик": если подряд зашло N человек 
  одного пола, следующий приоритет получает другой пол.
• Если представителей приоритетного пола нет, стрик сбрасывается.
• Выводится статистика: среднее время ожидания, загрузка, распределение по полу.

В отличие от многопоточной версии, здесь каждый студент — это **отдельный процесс**, 
а не поток. Это требует использования **разделяемой памяти** и **межпроцессной 
синхронизации**.

-------------------------------------------------------------------------------
2. КЛЮЧЕВЫЕ МЕХАНИЗМЫ ОПЕРАЦИОННОЙ СИСТЕМЫ
-------------------------------------------------------------------------------

Для реализации многопроцессной версии используются следующие системные вызовы:

=== 2.1 === shm_open() ===
#include <sys/mman.h>
#include <sys/stat.h>
#include <fcntl.h>

int shm_fd = shm_open("/bath_shm", O_CREAT | O_RDWR, 0666);

Назначение:
Создает или открывает POSIX-объект разделяемой памяти.

Параметры:
- name: "/bath_shm" — имя сегмента (глобальное в пределах системы)
- oflag: O_CREAT | O_RDWR
  - O_CREAT: создать, если не существует
  - O_RDWR: режим чтения и записи
- mode: 0666 — права доступа (чтение/запись для всех пользователей)

Возвращаемое значение:
Файловый дескриптор (shm_fd), который используется для дальнейших операций.

Особенности:
- Объект остается в системе даже после завершения программы (если не удалить)
- Должен быть явно удален через shm_unlink()

=== 2.2 === ftruncate() ===
#include <unistd.h>
#include <sys/types.h>

int ftruncate(int fd, off_t length);

Назначение:
Устанавливает точный размер объекта разделяемой памяти.

Почему нужно:
Новый сегмент изначально имеет размер 0. Нужно выделить место под структуру.

Пример:
ftruncate(shm_fd, sizeof(bath_t)) — устанавливает размер под структуру данных.

=== 2.3 === mmap() ===
#include <sys/mman.h>

void *mmap(void *addr, size_t length, int prot, int flags, int fd, off_t offset);

Назначение:
Отображает сегмент разделяемой памяти в адресное пространство процесса.

Параметры:
- addr: 0 — пусть ядро само выберет адрес
- length: sizeof(bath_t) — сколько байт отобразить
- prot: PROT_READ | PROT_WRITE — разрешить чтение и запись
- flags: MAP_SHARED — изменения видны другим процессам
- fd: shm_fd — файловый дескриптор сегмента
- offset: 0 — начать с начала

Возвращаемое значение:
Указатель на начало отображенной области. Все процессы получают одинаковый 
логический адрес (но физически может быть разный).

=== 2.4 === munmap() ===
int munmap(void *addr, size_t length);

Назначение:
Удаляет отображение разделяемой памяти из адресного пространства.

=== 2.5 === shm_unlink() ===
int shm_unlink(const char *name);

Назначение:
Удаляет именованный объект разделяемой памяти из системы.

Важно:
Вызывать в конце программы, чтобы не засорять систему.

=== 2.6 === fork() ===
#include <unistd.h>

pid_t fork(void);

Назначение:
Создает новый процесс — точную копию текущего.

Возвращаемое значение:
- В родительском процессе: PID дочернего процесса (> 0)
- В дочернем процессе: 0
- При ошибке: -1

Особенности:
- Дочерний процесс наследует все открытые дескрипторы, включая shm_fd
- Дочерний процесс начинает выполнение с того же места, где был вызван fork()
- У каждого процесса своя память, но разделяемая память общая

=== 2.7 === wait() ===
#include <sys/wait.h>

pid_t wait(int *status);

Назначение:
Родительский процесс блокируется до завершения любого дочернего процесса.

Используется в цикле:
for (int i = 0; i < TOTAL_STUDENTS; i++) {
    wait(NULL); // ждем всех студентов
}

-------------------------------------------------------------------------------
3. СИНХРОНИЗАЦИЯ МЕЖДУ ПРОЦЕССАМИ
-------------------------------------------------------------------------------

Так как несколько процессов одновременно работают с одними данными, нужна 
синхронизация. Используются pthread-примитивы, но с особыми атрибутами.

=== 3.1 === Атрибуты мьютекса: PTHREAD_PROCESS_SHARED ===

По умолчанию мьютексы работают только внутри одного процесса. Чтобы они 
работали между процессами, нужно установить атрибут:

pthread_mutexattr_t mattr;
pthread_mutexattr_init(&mattr);
pthread_mutexattr_setpshared(&mattr, PTHREAD_PROCESS_SHARED);

Аналогично для условных переменных:
pthread_condattr_setpshared(&cattr, PTHREAD_PROCESS_SHARED);

=== 3.2 === pthread_mutex_lock/unlock ===

Блокирует/разблокирует доступ к разделяемым данным. Только один процесс 
может держать мьютекс в каждый момент времени.

=== 3.3 === pthread_cond_wait() ===

- Освобождает мьютекс
- Переводит процесс в состояние ожидания
- Будится только при вызове pthread_cond_broadcast()

=== 3.4 === pthread_cond_broadcast() ===

Разбуживает **все** процессы, ожидающие на условной переменной. Необходимо, 
потому что после выхода студента в ванну может войти несколько других.

-------------------------------------------------------------------------------
4. СТРУКТУРА bath_t — ЦЕНТРАЛЬНЫЙ ОБЪЕКТ СИСТЕМЫ
-------------------------------------------------------------------------------

typedef struct {
    pthread_mutex_t mutex;            // основной мьютекс
    pthread_mutex_t mutex_inner;      // внутренний мьютекс для проверок
    pthread_cond_t cond;              // условная переменная для ожидания

    int capacity;                     // вместимость ванны
    int occupied;                     // сколько сейчас занято
    bath_state_t bath_gender;         // кто сейчас внутри (EMPTY, MEN_INSIDE, WOMEN_INSIDE)

    int streak_gender;                // кто имеет приоритет (-1 = не определён)
    int streak_used;                  // сколько подряд уже вошло
    int streak_limit;                 // максимальная длина стрика (по умолчанию 5)

    int remaining_male, remaining_female; // сколько мужчин/женщин ожидают

    long total_wait_time;             // суммарное время ожидания всех
    long total_bath_time;             // суммарное время пребывания
    int entered_count;                // всего вошло
    int male_entered, female_entered; // статистика по полу

    int verbose;                      // выводить ли детальные сообщения
} bath_t;

Важно: вся эта структура находится в разделяемой памяти, поэтому все процессы 
видят одни и те же данные.

-------------------------------------------------------------------------------
5. ЛОГИКА РАБОТЫ СТУДЕНТА (student_process)
-------------------------------------------------------------------------------

=== Шаг 1: Инициализация ===
- Генерация пола: случайно (50%/50%)
- Время мытья: от MIN_WASH_TIME до MAX_WASH_TIME секунд
- Засекается время начала ожидания

=== Шаг 2: Блокировка основного мьютекса ===
pthread_mutex_lock(&bath->mutex)

=== Шаг 3: Увеличение счетчика ожидающих ===
if (gender == MALE) bath->remaining_male++; else bath->remaining_female++;

=== Шаг 4: Цикл проверки возможности входа ===

while (1) {
    pthread_mutex_lock(&bath->mutex_inner);  // внутренняя проверка

    if (bath->occupied >= bath->capacity) {
        can_enter = 0;  // ванна переполнена
    }
    else if (bath->occupied == 0) {
        // Ванна пуста
        if (bath->streak_gender == -1) {
            can_enter = 1;  // любой может войти
        } else {
            int allowed = (bath->streak_used < bath->streak_limit) ?
                          bath->streak_gender : !bath->streak_gender;
            if (gender == allowed) {
                can_enter = 1;
            } else {
                // Сброс стрика, если приоритетного пола нет
                if ((allowed == MALE && bath->remaining_male == 0) ||
                    (allowed == FEMALE && bath->remaining_female == 0)) {
                    bath->streak_gender = -1;
                    bath->streak_used = 0;
                    can_enter = 1;
                }
            }
        }
    }
    else {
        // Ванна занята
        if ((bath->bath_gender == MEN_INSIDE && gender != MALE) ||
            (bath->bath_gender == WOMEN_INSIDE && gender != FEMALE)) {
            can_enter = 0;  // нельзя входить, если пол не совпадает
        } else {
            int allowed = (bath->streak_used < bath->streak_limit) ?
                          bath->streak_gender : !bath->streak_gender;
            can_enter = (gender == allowed);
        }
    }

    pthread_mutex_unlock(&bath->mutex_inner);

    if (can_enter) break;
    pthread_cond_wait(&bath->cond, &bath->mutex);  // ждем пробуждения
}

=== Шаг 5: Вход в ванну ===
- Обновление времени ожидания
- Увеличение bath->occupied
- Установка bath->bath_gender при первом входе
- Уменьшение счетчика ожидающих
- Обновление стрика
- Вывод сообщения "ВХОД"
- Разблокировка мьютекса

=== Шаг 6: Пребывание в ванне ===
sleep(wash_time);  // имитация времени мытья

=== Шаг 7: Выход из ванны ===
- Блокировка мьютекса
- Уменьшение bath->occupied
- Обновление статистики
- Если bath->occupied == 0, установка EMPTY
- Вывод "ВЫХОД"
- pthread_cond_broadcast(&bath->cond) — будим всех
- Разблокировка мьютекса

-------------------------------------------------------------------------------
6. СТРИК-МЕХАНИЗМ (FAIRNESS POLICY)
-------------------------------------------------------------------------------

Цель: предотвратить ситуацию, когда один пол постоянно занимает ванну.

Правила:
1. Если подряд вошло streak_limit студентов одного пола, следующий приоритет 
   переходит к другому полу.
2. Если представителей приоритетного пола нет в очереди, стрик сбрасывается.
3. При входе нового студента стрик обновляется:
   - Если пол совпадает — увеличиваем счетчик
   - Иначе — начинаем новый стрик

Пример:
- Заходят 5 мужчин → стрик достигнут
- Теперь приоритет у женщин
- Если женщины есть — они входят
- Если женщин нет — стрик сбрасывается, мужчины могут снова входить

7. ИНИЦИАЛИЗАЦИЯ И ЗАВЕРШЕНИЕ ПРОГРАММЫ
-------------------------------------------------------------------------------

=== main() — пошагово ===

1. shm_unlink("/bath_shm") — очищаем старую память
2. shm_open() + ftruncate() + mmap() — создаем и отображаем разделяемую память
3. bath_init() — инициализируем структуру и синхронизационные объекты
4. Цикл for: создаем TOTAL_STUDENTS процессов через fork()
   - Каждый процесс вызывает student_process()
5. Родитель ждет всех через wait()
6. Вывод статистики
7. Уничтожение мьютексов и условной переменной
8. munmap() + shm_unlink() — освобождение ресурсов

-------------------------------------------------------------------------------
8. ЧАСТЫЕ ОШИБКИ И КАК ИХ ИЗБЕЖАТЬ
-------------------------------------------------------------------------------

❌ Забыть shm_unlink() → утечка разделяемой памяти
✅ Всегда вызывать в конце программы

❌ Не использовать PTHREAD_PROCESS_SHARED → мьютексы не будут работать
✅ Всегда настраивать атрибуты

❌ Одинаковые rand() последовательности
✅ Использовать srand(time(NULL) ^ getpid())

❌ Неправильный порядок захвата мьютексов → взаимная блокировка
✅ Всегда захватывать в одном порядке: сначала mutex, потом mutex_inner

❌ Не ждать дочерние процессы → программа завершится раньше времени
✅ Использовать wait() в цикле

❌ Забыть ftruncate() → segfault при обращении к памяти
✅ Всегда устанавливать размер

11. ГЛУБОКОЕ ПОНИМАНИЕ СИСТЕМНОГО ВЫЗОВА fork()
-------------------------------------------------------------------------------

=== Что делает функция fork()? ===

fork() — системный вызов, который создаёт точную копию текущего процесса.

- Создаётся дочерний процесс, идентичный родительскому на момент вызова.
- Дочерний процесс получает:
  - Полная копия адресного пространства (кода, данных, стека, кучи)
  - Все открытые файловые дескрипторы (включая shm_fd!)
  - Текущее состояние регистров CPU
  - Текущий PID, PPID, приоритет, текущую директорию и т.д.

Но есть различия:
- У дочернего процесса свой уникальный PID
- Его PPID (родительский PID) равен PID родителя
- Некоторые ресурсы не копируются (например, блокировки мьютексов — но это сложная тема)

=== С какого места начинает работу дочерний процесс? ===

Дочерний процесс начинает выполнение сразу после вызова fork(), не с начала программы.

Пример:

    printf("До fork\n");
    pid_t pid = fork();
    printf("После fork\n");

Что выведется?
- В родителе: "До fork", затем "После fork" (один раз)
- В дочернем: "После fork" (один раз) — он не выводит "До fork" второй раз

Почему? Потому что fork() возвращает:
- В родителе: PID дочернего процесса (>0)
- В дочернем: 0
- При ошибке: -1

Поэтому можно определить, кто сейчас выполняется:

    if (pid == 0) {
        // Это код ДОЧЕРНЕГО процесса
    } else if (pid > 0) {
        // Это код РОДИТЕЛЬСКОГО процесса
    }

=== Что происходит после возврата из fork()? ===

После возврата из fork() оба процесса:
- Продолжают выполнение с той же строки, где был вызван fork()
- Имеют одинаковое состояние стека вызовов
- Идут по одному и тому же коду, но могут ветвиться через if (pid == 0)

Это называется развилка (fork) — отсюда и название функции.

=== Как вообще выполняется fork()? ===

На уровне ядра ОС:
1. Ядро выделяет новую запись в таблице процессов.
2. Копирует страницы памяти родителя (на практике — копирование при записи, copy-on-write).
3. Копирует дескрипторы файлов, сигналы, среду.
4. Устанавливает PPID дочернего процесса.
5. Запускает дочерний процесс, возвращая 0.
6. Родителю возвращается PID дочернего процесса.

Copy-on-write (COW):
- Страницы помечаются как "только для чтения"
- При попытке записи — возникает page fault → ядро делает копию страницы
- Экономит память и ускоряет fork()

12. ФУНКЦИЯ mmap() — КАК ОНА РАБОТАЕТ?
-------------------------------------------------------------------------------

mmap() — системный вызов, который отображает файл или устройство в виртуальное адресное пространство процесса.

Полный прототип:
void *mmap(void *addr, size_t length, int prot, int flags, int fd, off_t offset);

=== Параметры ===

addr: предпочтительный адрес в памяти (обычно 0 — пусть ядро выберет)
length: сколько байт отобразить
prot: права доступа (PROT_READ, PROT_WRITE, PROT_EXEC)
flags: режим отображения:
  MAP_SHARED — изменения видны другим процессам
  MAP_PRIVATE — изменения локальны
fd: файловый дескриптор объекта (например, shm_fd)
offset: смещение в объекте (обычно 0)

=== Что делает mmap()? ===

- Связывает участок виртуальной памяти процесса с физическим объектом
- После этого процесс может читать/писать в указанный адрес — данные автоматически 
  передаются в объект (разделяемая память, файл и т.п.)
- Не требует вызовов read()/write()

=== Почему mmap() эффективен? ===

- Доступ к данным как к обычной памяти — без накладных расходов системных вызовов
- Использует аппаратную поддержку MMU (Memory Management Unit)
- Подходит для больших данных и частых обращений

=== Особенности ===

- MAP_SHARED обязателен для межпроцессного взаимодействия
- mmap() работает только с объектами, поддерживающими отображение (файлы, shm)
- munmap() освобождает отображение

-------------------------------------------------------------------------------
13. ПОЧЕМУ РАЗДЕЛЯЕМАЯ ПАМЯТЬ БЫСТРАЯ?
-------------------------------------------------------------------------------

Разделяемая память — самый быстрый способ межпроцессного взаимодействия (IPC).

=== Причины высокой скорости ===

1. Нет копирования данных между процессами
   - В pipe, message queue данные копируются из одного адресного пространства в другое
   - В разделяемой памяти — один и тот же участок памяти используется напрямую

2. Прямой доступ по указателю
   - Чтение/запись как в обычную переменную
   - Нет системных вызовов для каждого доступа

3. Использование виртуальной памяти
   - Ядро отображает один и тот же физический RAM в разные процессы
   - Аппаратная поддержка MMU (Memory Management Unit)

4. Подходит для больших объемов данных
   - Передача массивов, структур, буферов — без накладных расходов

=== Минус: требуется синхронизация! ===

- Ядро не контролирует доступ
- Программист должен использовать мьютексы, семафоры и т.п.
- Иначе — состояние гонки (race condition)

-------------------------------------------------------------------------------
14. ЕСТЬ ЛИ ФУНКЦИЯ msync()? 
-------------------------------------------------------------------------------

Да, существует msync() — важно не путать!

Прототип:
int msync(void *addr, size_t length, int flags);

=== Назначение ===

Принудительно синхронизирует изменённые страницы в разделяемой памяти с физическим хранилищем (если оно есть).

=== Когда нужно? ===

- Если разделяемая память привязана к файлу на диске
- Чтобы гарантировать, что изменения записаны на диск
- Для отказоустойчивости

=== В нашем случае (POSIX shm) ===

msync() не обязателен, потому что:
- POSIX-объекты в памяти (/dev/shm/*) не сохраняются на диск
- Изменения теряются при перезагрузке
- Но это нормально для временных данных

=== Флаги ===

MS_SYNC — синхронная запись (блокируется до завершения)
MS_ASYNC — асинхронная (вернуть управление сразу)
MS_INVALIDATE — сбросить кэш страниц у других процессов

-------------------------------------------------------------------------------
15. СТРУКТУРА ПАМЯТИ ПРОЦЕССА
-------------------------------------------------------------------------------

Процесс имеет следующие сегменты памяти:

┌─────────────────┐ ← верхняя граница адресного пространства
│     Аргументы     │
│     и переменные │
│      окружения   │
├─────────────────┤ ← esp (stack pointer)
│       Стек       │ ← растёт вниз
│  (локальные переменные,│
│   параметры функций,   │
│   возвратные адреса)   │
├─────────────────┤
│                  │
│                  │
│       Пустота     │ ← heap growth
│                  │
├─────────────────┤ ← brk / sbrk (граница кучи)
│       Куча        │ ← растёт вверх (malloc, new)
│  (динамическая память) │
├─────────────────┤
│   Инициализированные │
│     данные (data)    │ ← глобальные переменные со значением
├─────────────────┤
│ Неиниц. данные (bss) │ ← глобальные переменные без значения (обнуляются)
├─────────────────┤
│     Код (text)     │ ← исполняемый код программы
└─────────────────┘ ← начало (0x400000 обычно)

=== Где лежат переменные? ===

Переменные объявленные внутри функции:
- Расположение: на стеке
- Пример: int x; внутри функции — на стеке

Переменные не объявленные внутри функции (глобальные):
- Расположение: в сегменте .data (если инициализированы) или .bss (если нет)
- Пример: int global_x = 10; → .data

Переменные, выделенные через malloc():
- Расположение: в куче
- Пример: int *p = malloc(sizeof(int)); → куча

Разделяемая память (через mmap):
- Расположение: отдельный регион (обычно выше кучи)
- Доступен всем процессам, которые его отобразили

-------------------------------------------------------------------------------
16. СТЕК ВЫЗОВОВ ФУНКЦИЙ — НЕ ТОЛЬКО СТРУКТУРА ДАННЫХ
-------------------------------------------------------------------------------

=== Что такое стек вызовов? ===

Не путать со структурой данных "стек". Здесь имеется в виду аппаратно-ориентированный стек процесса, реализованный на уровне CPU и ОС.

=== Назначение ===

- Хранение локальных переменных
- Хранение параметров функций
- Хранение возвратных адресов (куда вернуться после return)
- Сохранение контекста регистров при вызове функций

=== Регистры, связанные со стеком ===

esp (x86) / rsp (x86-64): Stack Pointer — указывает на вершину стека
ebp (x86) / rbp (x86-64): Base Pointer — указывает на базу текущего кадра стека

SS (Segment Selector): сегментный регистр, указывающий на сегмент стека в таблице дескрипторов

=== Как работает при вызове функции? ===

1. call function:
   - Автоматически помещает в стек адрес возврата
   - Переход на функцию
2. Начало функции:
   - push ebp — сохраняет старый base pointer
   - mov ebp, esp — устанавливает новый base pointer
3. Выделение локальных переменных:
   - sub esp, N — резервирует N байт на стеке
4. При выходе:
   - mov esp, ebp — восстанавливает esp
   - pop ebp — восстанавливает старый base pointer
   - ret — извлекает адрес возврата и переходит туда

=== Где находится стек? ===

- В выделенной области виртуальной памяти процесса
- Обычно располагается в верхней части адресного пространства
- Может быть несколько стеков (основной, потоковый, ядерный)

=== Особенности ===

- Размер ограничен (обычно 8 МБ по умолчанию в Linux)
- Переполнение стека → segmentation fault
- Используется для защиты (canaries, ASLR)

-------------------------------------------------------------------------------
17. ДОПОЛНЕНИЕ: С КАКОГО МЕСТА НАЧИНАЕТ РАБОТАТЬ ДОЧЕРНИЙ ПРОЦЕСС?
-------------------------------------------------------------------------------

Повторим ещё раз — это часто вызывает путаницу.

Дочерний процесс начинает выполнение сразу после возврата из fork().

Он:
- Не начинается с main()
- Не начинается с начала программы
- Продолжает выполнение с той же строки, где был вызван fork()
- Имеет тот же стек вызовов, те же локальные переменные, тот же контекст

Пример:

    void func() {
        printf("Внутри функции\n");
        pid_t pid = fork();  // ← здесь происходит fork
        printf("После fork\n");  // ← сюда придут ОБА процесса
    }

Оба процесса (родитель и дочерний) будут выполнять printf("После fork\n").

Различие только в значении pid:
- Родитель: pid > 0
- Дочерний: pid == 0

Поэтому можно делать ветвление:

    if (pid == 0) {
        // Только дочерний процесс выполнит это
        printf("Я — дочерний!\n");
        exit(0);
    }
    // Родитель продолжает дальше

-------------------------------------------------------------------------------
18. ЗАКЛЮЧЕНИЕ: КАК ВСЁ СВЯЗАНО?
-------------------------------------------------------------------------------

- fork() создаёт копию процесса → теперь два процесса имеют одинаковую память
- shm_open + mmap создают общий участок памяти → оба процесса могут его видеть
- pthread_mutexattr_setpshared(PTHREAD_PROCESS_SHARED) позволяет мьютексам работать между процессами
- mmap делает доступ к данным быстрым — как к обычной памяти
- msync нужен только если важна синхронизация с диском
- Стек хранит локальные переменные и управляет вызовами функций
- После fork() дочерний процесс продолжает с того же места — это ключ к пониманию

-------------------------------------------------------------------------------
19. ПОЧЕМУ В ЭТОЙ ПРОГРАММЕ НЕ ИСПОЛЬЗУЕТСЯ exec()? 
     fork() vs exec() — В ЧЁМ РАЗНИЦА?
-------------------------------------------------------------------------------

=== Краткий ответ ===

В этой программе используется только fork(), но не exec(), потому что:

► Нам не нужно запускать новую программу  
► Нам нужно, чтобы дочерние процессы выполняли тот же код, что и родитель  
► Мы хотим, чтобы все процессы-студенты работали по одному и тому же алгоритму  

Поэтому достаточно fork().  
exec() нужен только тогда, когда вы хотите **заменить образ процесса** на другой исполняемый файл.

---

=== Подробное объяснение: fork() и exec() — это разные инструменты ===

||| fork() |||
- Создаёт **копию текущего процесса**
- Дочерний процесс начинает выполнение с того же места (сразу после fork())
- Продолжает работать по тому же коду
- Используется для создания параллельных задач в рамках одной программы
- Пример: создание нескольких процессов-работников

||| exec() |||
- **Заменяет** текущий образ процесса на другой исполняемый файл
- Не создаёт нового процесса — перезаписывает текущий
- После вызова exec() старый код исчезает, начинает выполняться новый
- Возвращает управление **только при ошибке**
- Пример: запуск команды "ls", "grep" и т.п. из shell

---

=== Как они обычно используются вместе? ===

fork() и exec() часто идут **в паре**, особенно в shell:

    pid_t pid = fork();
    if (pid == 0) {
        // Дочерний процесс
        execl("/bin/ls", "ls", NULL);   ← заменяет этот процесс на /bin/ls
        perror("execl failed");         ← если вернулись — ошибка!
        exit(1);
    } else {
        // Родительский процесс
        wait(NULL);  // ждём завершения дочернего
    }

Этот паттерн называется **fork-exec**:
1. fork() — создаём копию
2. В дочернем: exec() — заменяем на новую программу
3. В родительском: ждём завершения или продолжаем работу

---

=== Почему в нашей программе exec() НЕ нужен? ===

Мы моделируем студентов, которые:
- Все делают одно и то же: пытаются войти в ванну, помыться, выйти
- Должны использовать один и тот же код (функция student_process)
- Разделяют данные через разделяемую память

Если бы мы сделали:

    pid_t pid = fork();
    if (pid == 0) {
        execl("./student_worker", "student_worker", id_str, NULL);
        exit(1);
    }

— это тоже сработало бы, **но усложнило бы систему**:
- Нужно компилировать отдельную программу `student_worker`
- Нужно передавать параметры через аргументы командной строки
- Сложнее отлаживать
- Нет преимущества в производительности

Вместо этого:
- Один бинарник
- Все процессы работают по одному коду
- Простая передача данных (через указатель bath)

---

=== Что происходит при вызове exec()? ===

При успешном вызове exec():
- Код процесса полностью заменяется
- Стек, куча, сегменты .text, .data — всё уничтожается
- Сохраняются:
  - Файловые дескрипторы (если не установлено FD_CLOEXEC)
  - PID, PPID
  - Рабочая директория
  - Переменные окружения
- Управление переходит к _start новой программы

---

=== Когда exec() действительно нужен? ===

1. Shell (bash, zsh): при вводе команды "gcc main.c"
   → fork() + exec("gcc")

2. Серверы, запускающие внешние скрипты
   → например, CGI в веб-сервере

3. Запуск сторонних утилит из своей программы
   → например, вызов "convert" из C-программы для обработки изображений

---

=== Вывод ===

► fork() — для создания параллельных процессов, выполняющих **тот же код**  
► exec() — для запуска **другой программы** внутри процесса  

Мы используем fork() без exec(), потому что:
- Все процессы-студенты должны выполнять одинаковую логику
- Нам не нужно загружать новый исполняемый файл
- Это проще, эффективнее и понятнее

Если бы студенты были реализованы как отдельные программы — тогда да, понадобился бы exec().
Но в нашем случае — достаточно одного вызова fork().

20. ПРОЦЕССЫ-ЗОМБИ: ЧТО ЭТО И ПОЧЕМУ ИХ НЕТ В НАШЕЙ ПРОГРАММЕ?
-------------------------------------------------------------------------------

=== Что такое процесс-зомби? ===

Процесс-зомби (zombie process) — это **завершившийся дочерний процесс, который всё ещё остаётся в таблице процессов ядра**, потому что родитель не считал его статус завершения.

- Процесс уже **не выполняется**
- Его код, память, стек — всё освобождено
- Но запись в таблице процессов остаётся
- Цель: чтобы родитель мог получить код завершения через wait()

Зомби потребляет только одну запись в таблице процессов — немного памяти, но если их много — может исчерпать системные ресурсы.

---

=== Как создаётся зомби? ===

Пример неправильного кода:

    pid_t pid = fork();
    if (pid == 0) {
        // Дочерний процесс
        printf("Я умираю\n");
        exit(0);  // завершается
    }
    // Родитель ничего не делает → не вызывает wait()

Результат:
- Дочерний процесс завершается
- Ядро помечает его как "зомби"
- Запись остаётся в таблице до тех пор, пока родитель не вызовет wait()

---

=== Почему в нашей программе нет зомби? ===

Потому что **родительский процесс явно ждёт всех дочерних**:

    for (int i = 0; i < TOTAL_STUDENTS; i++) {
        wait(NULL);
    }

Вызов wait():
- Блокируется, пока один из дочерних процессов не завершится
- Считывает статус завершения
- Удаляет запись о процессе из таблицы
- Освобождает ресурс → зомби не остаётся

---

=== Что будет, если не вызывать wait()? ===

- Все завершившиеся дочерние процессы станут зомби
- Они будут висеть в системе, пока:
  - Родитель не завершится
  - Или не вызовет wait()
- При завершении родителя все его зомби передаются процессу init (PID 1)
- init автоматически вызывает wait() и очищает их

Но полагаться на это — плохая практика.

---

=== Как избежать зомби? ===

1. Вызывать wait() после fork() — как мы делаем
2. Использовать сигнал SIGCHLD + обработчик + неблокирующий waitpid()
3. Игнорировать SIGCHLD: signal(SIGCHLD, SIG_IGN) — тогда зомби не создаются (зависит от системы)

В нашей программе используется способ №1 — простой и надёжный.

---

=== Проверить наличие зомби ===

Можно посмотреть в терминале:
    ps aux | grep 'Z'

Столбец STAT покажет "Z" для зомби.

---

=== Вывод ===

► Зомби — это "мертвые", но не похороненные процессы  
► Они не работают, но занимают место в таблице процессов  
► В нашей программе их нет, потому что родитель вызывает wait() для каждого  
► Это правильная практика при использовании fork()

-------------------------------------------------------------------------------
21. СЕМАФОРЫ — АЛЬТЕРНАТИВА МЬЮТЕКСАМ В МЕЖПРОЦЕССНОЙ СИНХРОНИЗАЦИИ
-------------------------------------------------------------------------------

=== Что такое семафор? ===

Семафор — это **целочисленный счётчик**, управляющий доступом к ресурсу, и набор атомарных операций для его изменения.

Автор: Эдсгер Дейкстра

Два основных типа:
- **Бинарный семафор**: значения 0 или 1 → работает как мьютекс
- **Счётный (счётниковый) семафор**: любое неотрицательное значение

---

=== Основные операции ===

||| P() (proberen — проверить) |||
- Уменьшает счётчик на 1
- Если счётчик < 0 → процесс блокируется
- Аналог: "захват"

||| V() (verhogen — увеличить) |||
- Увеличивает счётчик на 1
- Разблокирует один из ожидающих процессов
- Аналог: "освобождение"

В POSIX они называются:
- sem_wait() → P()
- sem_post() → V()

---

=== Чем семафор отличается от мьютекса? ===

||| Мьютекс |||
- Принадлежит потоку/процессу, который его захватил
- Только тот же процесс может разблокировать
- Поддерживает рекурсивный захват (в некоторых реализациях)
- Предназначен для взаимного исключения

||| Семафор |||
- Не имеет "владельца"
- Любой процесс может вызвать sem_post(), даже если не вызывал sem_wait()
- Может быть инициализирован на >1 — позволяет N процессам одновременно
- Подходит для управления доступом к пулу ресурсов

---

=== Пример: семафор вместо мьютекса в нашей задаче ===

Можно было бы использовать бинарный семафор:

    #include <semaphore.h>
    sem_t *mutex;

    // Инициализация в разделяемой памяти:
    mutex = mmap(...);
    sem_init(mutex, 1, 1);  // 1 = shared between processes, 1 = initial value

    // Вместо pthread_mutex_lock:
    sem_wait(mutex);

    // Вместо pthread_mutex_unlock:
    sem_post(mutex);

---

=== Когда использовать семафор, а когда мьютекс? ===

► Используйте **мьютекс**, если:
- Нужно защитить критическую секцию
- Только один процесс должен входить
- Тот же процесс блокирует и разблокирует

► Используйте **семафор**, если:
- Нужно ограничить доступ к N ресурсам (например, 3 кабинки)
- Один процесс производит, другой потребляет (producer-consumer)
- Нужна синхронизация между несвязанными процессами

---

=== Виды семафоров в Linux ===

||| Именованные семафоры (named semaphores) |||
- Имеют имя в файлоподобном пространстве: /dev/shm/sem.name
- Создаются через sem_open("/mysem", O_CREAT, 0666, 1)
- Видны всем процессам
- Переживают перезапуск процессов (до перезагрузки)

||| Безымянные семафоры (unnamed semaphores) |||
- Располагаются в разделяемой памяти или общей памяти
- Инициализируются через sem_init(&sem, pshared, value)
- pshared = 1 → для межпроцессного использования
- Уничтожаются через sem_destroy()

---

=== Почему в нашей программе используются мьютексы, а не семафоры? ===

Потому что:
- Мы реализуем **взаимное исключение** — классический случай для мьютекса
- Код был адаптирован из многопоточной версии (где используются pthread_mutex)
- Мьютексы лучше интегрированы с условными переменными (pthread_cond_wait)
- Логика "захватил-разблокировал" соответствует модели мьютекса

Но **семафоры тоже подошли бы** — выбор часто зависит от предпочтений и контекста.

---

=== Пример: счётный семафор для контроля вместимости ===

Вместо проверки bath->occupied вручную, можно использовать счётный семафор:

    sem_t *capacity_sem;
    sem_init(capacity_sem, 1, CAPACITY);  // разрешено CAPACITY процессов

    // При входе:
    sem_wait(capacity_sem);  // уменьшает счётчик
    // ... используем ресурс ...
    // При выходе:
    sem_post(capacity_sem);  // увеличивает счётчик

Это сделало бы управление вместимостью проще и надёжнее.

---

=== Вывод ===

► Семафор — мощный механизм синхронизации
► Подходит как для потоков, так и для процессов
► Гибче мьютекса, особенно при работе с пулами ресурсов
► В нашей программе можно было бы использовать, но мьютексы более естественны для текущей логики
